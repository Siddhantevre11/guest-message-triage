# Remove Source ID / HITL-for-missing-metadata; add a Host-facing frontend

Source ID and its HITL branch have been removed from the pipeline entirely — not made optional, not auto-generated, deleted. `check_source_id_node`, `hitl_node`, and the `source_id`/`hitl_triggered` state fields are gone. The graph now enters directly at the Orchestrator. Escalation remains as the pipeline's one human-handoff path (terminal, no pause-and-resume), for messages the Orchestrator flags as unclassifiable, that the Judge rejects three times, or where an LLM call exhausts its retries.

Source ID was originally introduced as provenance metadata to support a future booking-context lookup (RAG) — see ADR-0005, which already deferred that lookup out of scope. Requiring it now, and gating a HITL detour on a human supplying it, was premature: it solved a problem (linking a message back to its source for retrieval) that doesn't exist yet in this build, while adding a confusing step where a CLI operator was asked to type in "provenance metadata" as if they were the guest. It will be reintroduced when booking-context retrieval is actually built, attached automatically by whatever integration receives the message — not solicited from a human.

Separately, a minimal Flask-based frontend is introduced now rather than staying deferred (superseding part of ADR-0006). It is explicitly not production-hardened — no auth, no persistence beyond the existing JSONL run log. It presents a Host-inbox view: a Host submits a guest message and sees only the final triage result (category, summary, suggested action) as a clean card. The Orchestrator/Classifier/Judge reasoning trace, retries, and escalation mechanics stay server-side — visible in console output and `logs/triage.jsonl` for the developer, never rendered to the browser.
